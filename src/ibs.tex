\section{Стадии проектирования систем}

В процессе проектирование (информационной) системы выделяют следующие этапы: \cite[с. 53 -- 59 ]{NicholasSteyn_2008}
\subsection{Выявление требований и концептуальный дизайн}
Основной задачей данного этапа является выявление участников взаимодействия и их требований к системе. Анализ и построение концепции системы на высоком уровне.
    \subsubsection{Выявление заинтересованных лиц (stakeholders) и их потребностей.}
Создание структурированного списка необходимых функций ( высокоуровневых концепций ) с указанием лиц, заинтересованных в наличии данных функций. Обоснование потребности в каждой из заявленных функций. В процессе выяления требований аналитик также должен продумать систему критериев оценки эффективности проектирумого функционала, оценить необходимые затраты ресурсов и времени разработки.
    \subsubsection{Формулирование требований к системе.} Проводится систематизация высокоуровневых требований, выявленных на предыдущем этапе работ. Требования к системе должны быть четко сформулированы, должны быть согласованы с основными концептуальными аспектами разрабатываемой системы.\\ 
Требования должны быть:
        \begin{itemize}
            \item направлены на достижение задач, поставленных перед системой;
            \item соотнесены с этапами жизненного цикла системы;
            \item непротиворечивы основным ограничениям, накладываемым на систему;
            \item привязаны к интерфейсам взамодействия между элементами системы или интерфейсами с внешними системами.
        \end{itemize}
Полученный в результате список требований необходимо составить на языке, понятном заинтересованным лицам. Требования должны без излишних подробностей, четко выявлять суть требуемого функционала.
    \subsubsection{Оценка осуществимости.} Исходя из целей и задач, поставленных перед системой, в рамках сформулированных ограничений, необходимо оценить эффективность предложенного функционала для решения задач, достижения целей системы. Необходимо выявить и оценить альтернативные решения, способные привести систему в требуемое состояние, оценить риски для каждого из предложенных решений.  
    \subsubsection{Анализ требований к системе.} Детализация и формулирование списка функций системы необходимых и достаточных для удовлеворения потребностей ее пользователей. В отличие от предыдущего этапа, где требования к системе были рассмотрены с точки зрения пользователя, в рамках данного этапа требования уже рассматриваются, как предполагаемые функции проектируемой системы. В результате этапа должна быть сформирована функциональная спецификация системы.
    \subsubsection{Функциональные требования.} В функциональных требованиях должны быть отражен весь набор функций, необходимых пользователям, сформулированных на предыдущем этапе.
    \subsubsection{Требования возможности проверки и оценки произоводительности} Для качественной и количественной оценки каждому из сформулированных функциональных требований сопоставлен набор критерией оценки наличия заданных функций (качественный критерий) и покзателей эффективности(количественных).
    \subsubsection{Синтез спецификаций процессов} После проведения представленных работ необходимо выявить наличие уже реализованных продуктов, которые возможно было бы использовать в качестве модулей проектируемой системы. В качестве альтернативы рассмотреть возможность проектирования компонентов системы с нуля, учитывая сопутствующие риски.
