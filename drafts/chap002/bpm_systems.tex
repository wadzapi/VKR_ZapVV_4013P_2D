\subsection{Управление бизнес-процессами. Основные принципы и понятия}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
Исследуя сферу управления бизнес-процессами, вопросы терминологии и общепринятые практики обратимся к руководству BPM CBOK <<Свод знаний по управлению бизнес-процессами>> \cite{AbpmpCbok2_2009}, разработанному ABPMP (Association of Business Management Professionals). 

В терминологии ABPMP \cite[с. 24]{AbpmpCbok2_2009} под {\em управлением бизнес процессами} понимают диспциплинарный подход к деятельности по идентификации, разработке, выполнению, документированию, измерению, мониторингу, и контролю бизнес-процессов (автоматизированных и неавтоматизированных) для получения устойчивых результатов, согласованных со стратегическими целями органицации.

Жизненный цикл в практике управления бизнес-процессами, согласно ABPMP CBOK  \cite[с. 28]{AbpmpCbok2_2009}, предстваляет собой непревную циклическую последовательность из следующих деятельностей:
\begin{description}
  \item[Планирование] ---\, разработка стратегического подхода к организации процесса, задание целей и критериев оценки их достижения.
  \item[Анализ] ---\, проведение анализа текущей организации процессов на основе определенных целей и задач, оценка эффективности и влияния изменений на бизнес-среду организации.
  \item[Разработка и моделирование] ---\, разработка подхода к (ре)организации процесса, с проведением планирования ценности для потребителя. Моделирование процесса с учетом действующих в организации процессов, разработка подходов к интеграции с  компонентами организации, ее окружением. 
  \item[Реализация] ---\, внедрение процесса в структуру деятельности организации, действия по интеграции с действующими системами управления.
  \item[Мониторинг и контроль] ---\, деятельность по мониторингу выполнения процесса, оценка эффективности процесса с учетом разработанных критериев и стратегических целей.
  \item[Улучшение] ---\, выполнение новой итерации приведенных деятельностей с учетом полученных данных, пересмотр концепций и способа организации бизнес-деятельности.
\end{description}


\paragraph{Шаблонные практики управления процессами}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\label{bpms_usecases}
На основе анализа 289 научных работ Аалст в статье, посвященной десятилетию концепции управления бизнеc-процессами \cite{LNCS_Aalst_2012}, формулирует и выделяет следующие 20 вариантов использования из области управления бизнес-процессами:
\begin{description}
  \item[Разработка модели](design model, DesM) ---\, проектирование модели процессов с чистого листа;
  \item[Формулировка модели на основе данных о событиях] (discover model from event data, DiscM) ---\, автоматическая генерация модели из журнала событий; 
  \item[Выбор модели из коллекции] (select model for collection, SelM) ---\, выбор модели из репозитория готовых;
  \item[Объединение моделей] (merge models, MerM) ---\, синтез новой модели на основе слияния имеющихся;
  \item[Композиция моделей] (compose models, CompM) ---\, синтез новой модели составлением из компонентов имеющихся; 
  \item[Разработка шаблонной модели](design configurable model, DesCM) ---\, проектирование семейства моделей, зависящих от набора параметров;
  \item[Слияние моделелей в шаблон](merge models into configurable model, MerCM) ---\, объединение нескольких моделей в единую шаблонную модель с параметрами;
  \item[Настройка шаблонной модели](configure configurable model, ConCM) ---\, получение модели семейства на основе задания конкретных значений параметра;
  \item[Обновление модели](refine model, RefM) ---\, внесение изменений или перепроектирование модели процесса;
  \item[Принятие модели](enact model, EnM) ---\, получение одобрения и внедрение модели в деятельность организации;
  \item[Ведение журналов событий](log event data, LogED) ---\, регистрация значимых событий бизнес-модели в файлах журнала;
  \item[Мониторинг](monitor, Mon) ---\, отслеживание прогресса в выполнении заданных моделей деятельности.
  \item[Адаптирование при выполнении](adapt while running, AdaWR) ---\, изменение параметров, подстройка модели в режиме выполнения;
  \item[Анализ производительности на модели](analyze performance based on model, PerfM) ---\, деятельность по оценке производительности процесса с применением техник имитационного моделирования.
  \item[Проверка модели](verify model, VerM) ---\, формальные процедуры по выявлению конфликтов и ошибок в модели;
  \item[Проверка адекватности модели](check conformance using event data, ConfED) ---\, оценка адекватности модели на основе критерия согласованности с данными журнала событий;  
  \item[Анализ производительности](analyze performance using event data, PerfED)
  \item[Починка модели](repair model, RepM) ---\, внесение исправлений в модель процесса на основе анализа согласованности модели;
  \item[Расширение модели](extend model, ExtM) ---\, внесение в модель дополнительных данных, полученных из анализа событий;
  \item[Совершенствование модели](improve model, ImpM) ---\, пересмотр концепций модели с целью повышения показателей эффективности процессов.
\end{description}
Классификация деятельностей, опубликованная Аалстом, несет значительную ценность не только с теоретической, но и с практической точки зрения. Варианты использования характеризует состав набора деятельностей, выполняемых при управлении бизнес-процессами, и косвенно характеризуют состав функциональных требований к информационной системе по управлению бизнес-процессами.


\subsection{Информационные системы в управлении бизнес-процессами}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
Под системой упраления бизнес-процессов подразумевают целый комплекс программных средств и компонентов, направленных на реализацию и поддрежку деятельности по управлению бизнес-процесса на каждом из этапов его жизненного цикла. 
По определению в IT словаре крупной консалтинговой компании Gartner\cite{Gartner_bpms}, {\em cистемой управления бизнес-процессами} (в ориг. Business Process Management Suite), называют набор программного обеспечения для обеспечения и поддержки полного жизненного цикла программного обеспечения --- от выявления, формулирования, и проектирования процесса до реализации, мониторинга, анализа и последующей оптимизации. 
Схожим образом Ассоциация BPM-профессионалов определяет систему управления бизнес-процессами {\em (СУБП)}, как <<Программный комплекс, обеспечивающий моделирование, проектирование, разработку процессов и контролируемое выполнение работ и приложений. BPMS автоматически генерирует процессное приложение из процессных моделей и бизнес-правил, что позволяет осуществлять изменения очень быстро и под полным контролем.>>\cite{abpmp_bpms}


Авторами главы, посвященной вопросам моделирования и управления бизнес-процессами, книги <<Руководство по инормационным системам в бизнесе>> (Handbook on Business Information Systems) \cite{ElMekawyShahzad_2010} были выделены группы функциональных возможностей, свойственных инфорационным системам управления бизнес-процессами.
Среди функций информационных систем управления бизнес-процессами на основе анализа и группировки по стадиям жизненного цикла выделяют следующие группы: \cite[с. 100 -- 101]{ElMekawyShahzad_2010}
\begin{description}
  \item[Задание процессов] ---\, функции, связанные с моделированием и разработкой бизнес-процесса;
  \item[Хранение процессов] ---\, функции обеспечения хранения и администрирования бизнес-моделей;
  \item[Изменение процессов] ---\, функции обеспечения возможностей добавления, изменения, удаления и извлечения моделей;
  \item[Межмодельные отношения] ---\, задание и обоснование связей между молелями процесса и бизнес-деятельности;
  \item[Задание цели] ---\, функции целеполагания, обеспечение возможностей задания стратегической цели бизнес-деятельности;
  \item[Соотнесения целей и процессов] ---\, соотнесение определений процессов с решаемыми задачами и установленными целями; 
  \item[Имитационное моделирование] ---\, обеспечение возможностей динамического моделирования заданных процессов;
  \item[Исполнение процессов] ---\, обеспечение выполнения процессов бизнес-деятельностей; 
  \item[Мониторинг и оценка производительности] ---\, обеспечение возможностей мониторинга, оценки эффективности процессов;
  \item[Анализ процессов] ---\, обеспечение необходимыми критериями, показателями и функциями по анализу процессов;
\end{description}

\subsection{Шаблоны компонентов модели потоков работ}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
{\em }
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\label{workflow_model_patterns}
Аалст, Хофстереде и другие исследователи, принимавшие участие в разработке языка YAWL, названного с долей иронии, "очередным языком управления работ", выбрали вполне разумный подход к процессу сбора функциональных требований --- \, на основе опыта аналогичных программ. Задачей исследования было выделение основных шаблонов в моделировании процессов и формирование из данного набора шаблонов модельной деятельности необходимого и достаточного (без избыточной сложности) набора необходимых функций. В научной статье о шаблонах workflow \cite{AalstHofstede_2003} был опубликован список из 20-ти выделенных шаблонах моделей и их компонентов. 

При выборе кандидатур в список шаблонов моделения Аалст и Хофстеде ориентировались на количественные показатели, оценивая частоту встречаемости функционала в программах из материала анализа. Считая залогом "шаблонности" наличие функционала рассматриваемых системах, для полносты описания шаблон сопровождается описанием, синонимами и примерами из практики. Среди всех функциональных возможностей систем моделирования потоков Аалст и Хофстеде в работе \cite[с. 10 -- 39]{AalstHofstede_2003} выделяют следующие:
\begin{description}
  \item[Последовательность деятельностей] (Sequence) ---\, соединение, когда каждая последующая деятельность в потоке начинается после завершения предшествующей.
  \item[Ветвление потока] (Parrallel split) ---\, точка разделения единого потока управления на несколько, выполняемых параллельно.  
  \item[Синхронизация потоков] (Syncronization) ---\, точка слияния нескольких параллельных подпроцессов в единый поток управления. 
  \item[Принятме решения] (Exclusive choice) ---\, точка в потоке работ, где ветвление происходит на основе принятого решения. 
  \item[Простое слияние] (Simple merge) ---\, точка в которой несколько альтенетивных ветвей могут быть соединены в единый поток без синхронизации. 
  \item[Мультивыбор] (Multi-choice) ---\, точка потока работ, в которой на основе принятого решения или данных, постурающих от управления, производится выбор некоторого набора ветвей. 
  \item[Слияние с синхронизацией] (Synchronizing merge) ---\,  необходимость слияния нескольких параллельных ветвей возможного протекация потока работ (после мультивыбора), синхронизация выполняется для всей группы активных подпотоков.  
  \item[Мультислияние] (Multi-merge) ---\, точка потока, где две или более параллельные ветви сходятся в единый поток без синхронизации. 
  \item[Определитель] (Disctiminator) ---\, точка потока, в которой выполнение последующих действий начинается после завершения по крайней мере одной входной деятельности. 
  \item[Зацикливание] (Arbitrary cycles) ---\, точка в потоке, где одна или несколько деятельностей выполняться циклически.  
  \item[Неявное завершение] (Implicit termination) ---\, каждый из подроцессов потока работ должен быть завершен, если в его составе не осталость активных деятельностей, и если ни одна из деятельностей не может быть активизирована. 
  \item[Множественность экземпляров без синхронизации] (Multiple instances without syncronization) ---\, возможность создания в режиме выполнения для одного потока нескольких экземпляров деятельности без необходимости в последующей синхронизации. 
  \item[Множественность экземпляров, известная на этапе проектирования] (Multiple instances with a priori design time knowledge) ---\, создание нескольких экземпляров деятельности для одного потока работ, с заранее известным на этапе проектирования числом необходимых экземпляров. 
  \item[Множественность экземпляров, известная на этапе выполения] (Multiple instances with a priori runtime knowledge) ---\, создание нескольких экземляров деятельности в рамках потока, с числом экземпляров, известным на заранее, на одном из этапов выполнения. 
  \item[Множественность, не известная на этапе выполнения] (Multiple instances without a priory runtime knowledge) ---\, создание нескольких экземпляров деятельности внутри потока, причем число их неизвестно ни на этапе проектирования, ни на этапе выполнения.
  \item[Отложенный выбор] (Deffered choice) ---\, точка в процессе, где выбор производится из множества возмжных вариантов ветвей, причем выбирается только одна из них. Отличительными особенностями данного шаблона является неопределенность выбора и стремление откладывания решения как можно дольше.  
  \item [Поочередное параллельное выполнение] (Interleaved parallel routing) ---\, набор деятельностей, выполняемых поочередно, причем поряд очередности выбирается в режиме выполнения, в каждым момент времени выполняется только один подпоток.
  \item[Контрольная точка] (Milestone) ---\, активизация деятельностей зависит от состояния, в котором находится поток работ на момент достижения заданной точки. 
  \item[Отмена деятельности] (Cancel activity) ---\, деактивизация деятельности, например в момент ожидания заврешения другой деятельности. 
  \item[Отмена экземляра потока] (Cancel case) ---\, полная отмена и прекращение всех деятельностей, связанных с экземпляром потока. 
\end{description}  

Стоит отдельно подчеркнуть, что в исследовании шаблон моделирования применяется скорее в смысле инструмента, предназначенного в большей мере не для формирования самой модели, а направленного на задание состава компонентов, инструментария системы моделирования. Таким образом, можно утверждать, что представленные компоненты являются в большей степени функциональным каркасом (framework) для проектирования языков и нотаций потоков работ.
